# Pocket-T Audit Report (2026-05-22)

## Scope

This audit covered repository structure, build/test health, and high-risk security/operational paths for `pt-registry` and workspace packaging.

## Commands Executed

- `pnpm -r test`
- `pnpm -r build`
- `pnpm audit --prod`
- targeted source review in daemon and install/runtime paths

## Executive Summary

- **Build status:** Workspace builds successfully.
- **Test status:** One daemon suite fails in fresh state due to unresolved `@pocket-t/shared` package entry.
- **Security posture:** LAN/default-local boundary is clear, but control surfaces currently allow unauthenticated browser/WS access on whichever host the daemon binds; this is acceptable only if loopback-only is preserved.
- **Operational reliability:** Good resilience patterns (relay reconnect/backoff, detach grace). A few defaults can surprise users in edge cases.

## Findings

### 1) Test-order dependency / packaging mismatch (Medium)

`pnpm -r test` fails in `packages/daemon/src/ws-v3.test.ts` because Vite/Vitest cannot resolve `@pocket-t/shared` entrypoint in a clean test run. This indicates test setup depends on prior build artifacts or export-map mismatch.

Impact:
- CI flakes or local contributor friction.
- Hidden coupling between package build order and test execution.

Recommendation:
- Ensure `@pocket-t/shared` has valid `exports` / `main` / `types` for source-mode testing, or
- Add explicit pretest build step for shared package in daemon test pipeline, or
- Configure Vitest alias to shared source (`packages/shared/src/index.ts`) for tests.

### 2) Unauthenticated browser control channel if host is exposed (High if misconfigured, Low on default loopback)

Browser clients are attached over `/ws` with no auth/token/origin gate in server-side connection flow. If `POCKET_T_BROWSER_HOST` is changed to `0.0.0.0` (or proxied externally), anyone reaching the port could enumerate sessions and inject input.

Impact:
- Full interactive terminal control by network-reachable clients.

Recommendation:
- Add optional required auth token (header/query) for `/ws` and HTML endpoint.
- Add explicit origin allowlist checks.
- Print strong runtime warning when host is not loopback and auth is disabled.

### 3) Approval hook default bypass when no viable listener (Design risk)

Hook server uses `defaultOnNoListener: 'approve'` and intentionally bypasses blocking when no viable UI listener is present. This is documented in-code rationale, but it shifts to "allow-by-default" behavior during transient UI disconnects.

Impact:
- Tool approval UX expectations may differ from user intent in disconnected periods.

Recommendation:
- Keep current behavior for ergonomics, but expose an explicit strict mode env var (e.g., `POCKET_T_APPROVAL_STRICT=1`) to fail closed.

### 4) Dependency vulnerability scan currently blocked by registry response (Low)

`pnpm audit --prod` returned HTTP 403 from npm audit endpoint in this environment; no vulnerability inventory could be produced.

Impact:
- Unknown dependency CVE status from this run.

Recommendation:
- Run audit in CI with allowed endpoint access, or use an alternate scanner (e.g., OSV/Snyk/GitHub Advisory) as a fallback.

## Positive Observations

- Good recovery/backoff behavior in relay client reconnect logic.
- Session detach grace handling improves UX and robustness.
- Build pipeline succeeds across shared/daemon/relay packages.

## Priority Actions

1. Fix daemon test import resolution so `pnpm -r test` passes from clean checkout.
2. Add auth gates for non-loopback browser server exposure.
3. Add strict approval mode toggle for security-sensitive users.
4. Restore automated dependency audit coverage in CI.
